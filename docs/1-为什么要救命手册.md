# 1. 为什么要救命手册？

## 🎭 三个真实故事

### 故事一：新窗口失忆症

> **周一**，我和 Claude 一起做一个 Go + Qdrant 项目，Phase 2 三段筛选管线跑通了，部署到远程服务器，爬虫数据实时进来。**开心收工。**
>
> **周二早上**，新开 Claude 窗口继续：
>
> - 我："继续做昨天的项目"
> - Claude："好的，请告诉我项目在哪里？"
> - 我：花了 **30 分钟**解释项目背景、技术栈、昨天做到哪一步...
> - Claude："好的，所以你是想做一个...（重复我的话）"
> - 我：抓狂 😤

**问题根源**：Claude 不是同一个人。每个新窗口都是**失忆的新员工**。

**解决方案**：写 `ONBOARDING.md`，新 Claude 30 秒读完就能进入状态。

---

### 故事二：代码丢了 ≠ 代码没了

> 某个项目没建 git。**某次误操作**，代码文件被覆盖。
>
> 我问 Claude："我们之前不是实现过 X 功能吗？"
>
> Claude（通过记忆系统）："对，做过的，你看当时的进展记录是 ..."
>
> 我去找代码："**代码呢？**"
>
> ...找不到。记忆系统记了"Phase 2 已跑通"，但实际代码**从未保存到 git**。运行中的 binary 在服务器上，但源码已经丢了。

**问题根源**：
- 记忆系统记录的是"进度叙事"，不是代码本身
- 没有 git 就没有快照，本地一改就丢

**解决方案**：
- `STATUS.md` 和 git 都要更新，才构成**证据链**
- 每次会话结束 `git commit + push`（`OFFBOARDING.md` 第 3 步）

---

### 故事三：密钥推到公网

> 项目着急上线，Claude 帮我把代码推到 GitHub。没 `.gitignore`，直接 `git add .`。
>
> **配置文件里的 OpenAI key 推到 public repo 了。**
>
> 10 分钟后，key 被爬虫扫到，被盗刷 ¥XXX。

**问题根源**：
- `git add .` 无差别添加
- `.gitignore` 没配置好
- 密钥没分离到独立文件

**解决方案**：
- `.gitignore.tpl` 先铺好再 `git init`
- 每次 commit 前 `git status --ignored --short` 确认
- 密钥三位一体：`config.yaml`（不入库）+ `config.example.yaml`（入库）+ `credentials.md`（本地）

## 🧠 心智模型：项目是有"生命"的

每个项目都有生命周期：

```
诞生 → 快速试错 → 原型稳定 → 投产运行 → 偶尔维护 → 归档 / 复活
```

**每个阶段都会留下债务**：
- 没记录的决策
- 没 commit 的改动
- 没写进文档的陷阱
- 没备份的数据
- 没测试的操作

**债务不会自动消失**，只会在未来某个"我要救命"的时刻集中爆发。

救命手册的核心理念：**每次会话结束时还一点债**，不让它累积到救不回来。

---

## 🤝 非程序员的独特处境

程序员习惯了版本控制、部署自动化、故障排查。**你不是。**

当 Claude 和非程序员合作时：
- 你不会盯 git log 看
- 你不会每周备份数据库
- 你不会手动写 deploy 脚本
- 你不会在脑子里装 10 个服务器 IP 和端口

所以 Claude 必须**帮你建一套自动保护**，不靠你记，靠文档。

这就是本仓库的意义。

---

## ✅ 你会得到什么

用这套方法论的项目：

- ✅ 任何新 Claude 60 秒进入状态
- ✅ 下班前自动存档，不会漏
- ✅ 电脑丢了，GitHub + 记忆系统能恢复
- ✅ 密钥不会推公网（`.gitignore` + 三位一体）
- ✅ 服务挂了，RUNBOOK 查 1 分钟有答案
- ✅ 6 个月后重新拾起，STATUS.md 告诉你到哪一步了

**代价**：一次性 30 分钟建立。以后每次会话结束多花 2 分钟 offboard。

**收益**：**永不失联**。
