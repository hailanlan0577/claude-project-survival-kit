# 3. 密钥管理三位一体

> **最常见的灾难**：把密钥推到 GitHub public repo，被爬虫扫到盗刷。
>
> 这一篇教你怎么建立"**密钥永不入库、随时能恢复**"的三位一体布局。

---

## 🎯 三位一体：三个文件各司其职

```
┌─────────────────────────────────────────────┐
│ 1. configs/config.yaml (或 .env)            │
│    ↓                                         │
│    真实密钥，程序运行时读                    │
│    🚫 .gitignore 严格排除                    │
│    📦 只存在本地硬盘 + 部署目标              │
└─────────────────────────────────────────────┘

┌─────────────────────────────────────────────┐
│ 2. configs/config.example.yaml              │
│    ↓                                         │
│    脱敏模板，保留结构，值用占位符            │
│    ✅ 入 git 仓库                            │
│    📖 告诉未来的人：这个项目需要哪些密钥     │
└─────────────────────────────────────────────┘

┌─────────────────────────────────────────────┐
│ 3. ~/.claude/projects/*/memory/             │
│    credentials.md                           │
│    ↓                                         │
│    全项目密钥总表，本地唯一副本              │
│    🔒 不在 git 里（个人记忆系统）            │
│    🆘 config.yaml 丢了能从这里拼回          │
└─────────────────────────────────────────────┘
```

---

## 📄 三个文件具体长啥样

### 1. `configs/config.yaml`（真实密钥）

```yaml
# 本文件已被 .gitignore 排除，永不入库

dashscope:
  api_key: "sk-0d..."  # ← 真实 key

feishu:
  app_id: "cli_xxx"
  app_secret: "LEHUddPp..."
  relay_chat_id: "oc_xxx..."
```

### 2. `configs/config.example.yaml`（脱敏入库）

```yaml
# 复制到 configs/config.yaml 并填真实值
# configs/config.yaml 已 gitignore

dashscope:
  api_key: "sk-xxxxxxxxxxxx"  # 从阿里云 DashScope 控制台获取

feishu:
  app_id: "cli_xxx"              # 开放平台应用 App ID
  app_secret: "xxxxxxx"          # 应用 Secret（保密）
  relay_chat_id: "oc_xxx"        # 数据中转群 chat_id，/open-apis/im/v1/chats 查
```

**关键**：字段全保留 + 注释说明"去哪拿真实值"。

### 3. `~/.claude/projects/*/memory/credentials.md`

```markdown
## 项目 X 完整密钥

> 权威副本：本地 + 部署目标的 configs/config.yaml
> 两地均在 .gitignore，永不入库

### DashScope
- api_key: `sk-0d...`

### 飞书
- app_id: `cli_xxx`
- app_secret: `LEHU...`
- relay_chat_id: `oc_xxx`

...（其他所有密钥）

### 恢复方式
如果两地 config.yaml 全丢：按上表拼回 config.yaml，
参考仓库 configs/config.example.yaml 的结构。
```

## 🔍 怎么验证密钥不会泄露

### 1. `.gitignore` 正确配置

```
# 必须
configs/config.yaml
configs/*.key
configs/*.pem
.env
.env.local
.env.*.local
```

### 2. 每次 commit 前扫一眼

```bash
# 看 staged 内容有没有敏感文字
git diff --cached | grep -iE "api_key|secret|password|token"

# 看 ignored 列表里有没有敏感文件
git status --ignored --short | grep '!!' | grep -E 'config\.yaml|\.env|\.key'
```

### 3. push 前最后确认

```bash
git log origin/main..HEAD --stat
# 看看要推的 commits 里有没有 configs/config.yaml 等危险文件
```

## 🚨 如果已经推上 GitHub 了怎么办

**按 3 步抢救**（RUNBOOK.md § 10.9 详细版）：

### 第 1 步：立刻轮换所有泄露的密钥

| 平台 | 去哪轮换 |
|------|---------|
| OpenAI / Anthropic | Console 里删旧 key 创建新 key |
| 阿里云 DashScope | 控制台 → API-KEY 管理 |
| 飞书 | 开放平台 → 应用管理 → 凭证 → 重置 |
| 钉钉 | 群 → 机器人 → 移除重建 |
| 企业微信 | 企业微信后台 → 应用 → 重置 secret |
| GitHub PAT | `github.com/settings/tokens` → Revoke |

**即使看起来没人看到也必须轮换**—— private repo 也可能被内部泄露，public repo 更是 10 分钟内就会被扫到。

### 第 2 步：从 git 历史抠掉

```bash
brew install git-filter-repo
cd <项目>
git filter-repo --path configs/config.yaml --invert-paths --force
git push --force  # 覆盖远程旧历史
```

### 第 3 步：验证清理

```bash
git log --all --full-history -- configs/config.yaml
# 应该啥都没有

# GitHub 网页翻 commits，确认找不到
```

## 💡 三位一体的优势

| 场景 | 结果 |
|------|------|
| 本地硬盘坏了 | credentials.md + GitHub 的 example 拼回 |
| 部署目标坏了 | 本地 config.yaml + credentials.md 恢复 |
| 不小心 `git add .` | `.gitignore` 拦住，config.yaml 不入 staging |
| 换新机器 | clone repo + 按 credentials.md 填 config.yaml |
| Claude 帮我改密钥 | 三地都要改（本地 / 部署 / credentials.md） |
| 想给朋友 fork 项目 | 他拿到 example，自己填 config.yaml |

**没有任何场景会让密钥泄露或丢失**。

## ⚠️ 反模式（别这么做）

### ❌ 把密钥放 README.md

```markdown
## 如何运行

需要 OpenAI key: sk-0d...
```

**永远不要**。README 会被搜索引擎索引、被 fork、被 mirror。

### ❌ hard-code 在源代码里

```python
client = OpenAI(api_key="sk-0d...")  # ❌
```

### ❌ 只在 `.env.example` 里写字段名但没说去哪拿

```
OPENAI_API_KEY=
FEISHU_APP_ID=
```

这样 6 个月后的自己也不记得该填啥。**注释写清楚去哪拿真实值**。

### ❌ credentials.md 推到 GitHub

credentials.md 是**本地独有**的备份。推到 GitHub 就失去意义了。
